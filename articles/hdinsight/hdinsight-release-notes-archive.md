# <a name="archived-release-notes"></a>Archivované poznámky k verzi

> [!IMPORTANT]  
> HDInsight od verze 3.4 výše používá výhradně operační systém Linux. Další informace naleznete v [článku správy verzí HDInsight](hdinsight-component-versioning.md).

## <a name="summary"></a>Souhrn

Azure HDInsight je jednou z nejoblíbenějších služeb mezi podnikovými zákazníky pro analýzy Apache Hadoop a Apache Spark s otevřeným zdrojovým kódem v Azure.

### <a name="new-features"></a>Nové funkce

#### <a name="service-tags"></a>Značky služeb
Značky služeb zjednodušují zabezpečení virtuálních počítačů Azure a virtuálních sítí Azure tím, že umožňují snadno omezit přístup k síti ke službám Azure. Značky služeb v pravidlech skupiny zabezpečení sítě (NSG) můžete použít k povolení nebo odepření provozu na konkrétní službu Azure globálně nebo v oblasti Azure. Azure poskytuje údržbu IP adres, které jsou základem každé značky. Značky služeb HDInsight pro skupiny zabezpečení sítě (NSG) jsou skupiny IP adres pro služby stavu a správy. Tyto skupiny pomáhají minimalizovat složitost při vytváření pravidel zabezpečení. Zákazníci HDInsight můžou povolit značku služeb prostřednictvím Azure Portal, PowerShellu a ROZHRANÍ REST API. Další informace najdete v [tématu značky služeb skupiny zabezpečení sítě (NSG) pro Azure HDInsight](https://docs.microsoft.com/azure/hdinsight/hdinsight-service-tags).

#### <a name="custom-ambari-db"></a>Vlastní Ambari DB

Tato funkce umožňuje zvolit správnou databázi SQL DB pro vaše potřeby zpracování a kapacity. Můžete také snadno upgradovat tak, aby odpovídaly požadavkům na růst podniku. Další informace naleznete v tématu [Nastavení clusterů HDInsight pomocí vlastní ambariské databáze](hdinsight-custom-ambari-db.md).

#### <a name="transport-layer-security-tls-12-enforcement"></a>Zabezpečení transportní vrstvy (TLS) 1.2 vynucení
Zabezpečení transportní vrstvy (TLS) a SSL (Secure Sockets Layer) jsou kryptografické protokoly, které poskytují zabezpečení komunikace v počítačové síti. Další informace naleznete v tématu [Transport Layer Security](https://en.wikipedia.org/wiki/Transport_Layer_Security#SSL_1.0.2C_2.0_and_3.0). Zatímco clustery Azure HDInsight přijímají připojení TLS 1.2 na veřejných koncových bodech HTTPS, TLS 1.1 je stále podporován pro zpětnou kompatibilitu se staršími klienty.

Počínaje další verzí budete moci přihlásit a nakonfigurovat nové clustery HDInsight tak, aby přijímala pouze připojení TLS 1.2. 

Později v roce, počínaje 6/30/2020, Azure HDInsight vynutí TLS 1.2 nebo novější verze pro všechna připojení HTTPS. Doporučujeme zajistit, aby všichni vaši klienti byli připraveni ke zpracování verzí TLS 1.2 nebo novějších verzí.

#### <a name="moving-to-azure-virtual-machine-scale-sets"></a>Přechod na škálovací sady virtuálních strojů Azure
HDInsight teď používá virtuální počítače Azure ke zřizování clusteru. Počínaje únorem 2020 (přesné datum bude sděleno později), HDInsight místo toho použije škálovací sady virtuálních strojů Azure. Další informace o [škálovacích sadách virtuálních strojů Azure](https://docs.microsoft.com/azure/virtual-machine-scale-sets/overview).

HDInsight Identity Broker (HIB) umožňuje uživatelům přihlásit se k Apache Ambari pomocí vícefaktorového ověřování (MFA) a získat požadované lístky protokolu Kerberos bez nutnosti zahánění hodnot hesel ve službě Azure Active Directory Domain Services (AAD-DS). V současné době je hib k dispozici jenom pro clustery nasazené prostřednictvím šablony Správy prostředků Azure (ARM).

#### <a name="auto-scale"></a>Automatické škálování

Automatické škálování pro Azure HDInsight je teď obecně dostupné ve všech oblastech pro typy clusterů Apache Spark a Hadoop. Tato funkce umožňuje spravovat úlohy analýzy velkých objemů dat nákladově efektivnějším a produktivnějším způsobem. Nyní můžete optimalizovat využití clusterů HDInsight a platit pouze za to, co potřebujete.

V závislosti na vašich požadavcích si můžete vybrat mezi automatickým škálováním založeným na zatížení a na plánu. Automatické škálování založené na zatížení může škálovat velikost clusteru nahoru a dolů na základě aktuálních potřeb prostředků, zatímco automatické škálování založené na plánu může změnit velikost clusteru na základě předdefinovaného plánu. 

Podpora automatického škálování pro úlohy HBase a LLAP je také veřejná preview. Další informace najdete [v tématu Automatické škálování clusterů Azure HDInsight](https://docs.microsoft.com/azure/hdinsight/hdinsight-autoscale-clusters).

### <a name="known-issues"></a>Známé problémy

#### <a name="replication-is-broken-for-secure-hbase-with-default-installation"></a>Replikace je přerušena pro Secure HBase s výchozí instalací

V případě rozhraní HDInsight 4.0 postupujte takto:

1. Povolte komunikaci mezi clustery.
1. Přihlaste se k aktivnímu headnode.
1. Stáhněte si skript, který povolí replikaci pomocí následujícího příkazu:

    ```
    sudo wget https://raw.githubusercontent.com/Azure/hbase-utils/master/replication/hdi_enable_replication.sh
    ```
1. Zadejte příkaz `sudo kinit <domainuser>`.
1. Chcete-li spustit skript, zadejte následující příkaz:

    ```
    sudo bash hdi_enable_replication.sh -m <hn0> -s <srclusterdns> -d <dstclusterdns> -sp <srcclusterpasswd> -dp <dstclusterpasswd> -copydata
    ```
V případě hdinsightu 3.6 postupujte takto:

1. Přihlaste se k aktivnímu HMaster ZK.
1. Stáhněte si skript, který povolí replikaci pomocí následujícího příkazu:
    ```
    sudo wget https://raw.githubusercontent.com/Azure/hbase-utils/master/replication/hdi_enable_replication.sh
    ```
1. Zadejte příkaz `sudo kinit -k -t /etc/security/keytabs/hbase.service.keytab hbase/<FQDN>@<DOMAIN>`.
1. Zadejte následující příkaz:

    ```bash
    sudo bash hdi_enable_replication.sh -s <srclusterdns> -d <dstclusterdns> -sp <srcclusterpasswd> -dp <dstclusterpasswd> -copydata
    ```

#### <a name="phoenix-sqlline-stops-working-after-migrating-hbase-cluster-to-hdinsight-40"></a>Phoenix Sqlline přestane fungovat po migraci clusteru HBase do HDInsight 4.0

Postupujte takto:

1. Přetáhněte následující tabulky Phoenix:
    1. `SYSTEM.FUNCTION`
    1. `SYSTEM.SEQUENCE`
    1. `SYSTEM.STATS`
    1. `SYSTEM.MUTEX`
    1. `SYSTEM.CATALOG`
1. Pokud nemůžete odstranit žádnou z tabulek, restartujte HBase vymazat všechna připojení k tabulkám.
1. Spusťte `sqlline.py` znovu. Phoenix znovu vytvoří všechny tabulky, které byly odstraněny v kroku 1.
1. Regenerujte phoenixské tabulky a zobrazení pro vaše data HBase.

#### <a name="phoenix-sqlline-stops-working-after-replicating-hbase-phoenix-metadata-from-hdinsight-36-to-40"></a>Phoenix Sqlline přestane pracovat po replikaci metadat HBase Phoenix z HDInsight 3.6 na 4.0

Postupujte takto:

1. Před provedením replikace přejděte do cílového clusteru 4.0 a spusťte . `sqlline.py` Tento příkaz bude generovat `SYSTEM.MUTEX` `SYSTEM.LOG` Phoenix tabulky jako a že existují pouze v 4.0.
1. Přetáhněte následující tabulky:
    1. `SYSTEM.FUNCTION`
    1. `SYSTEM.SEQUENCE`
    1. `SYSTEM.STATS`
    1. `SYSTEM.CATALOG`
1. Zahájení replikace HBase

#### <a name="mahout"></a>Mahout

V HDP-2.3.x a 2.4.x jsme místo odeslání konkrétního apache verze Mahoutu synchronizovali s konkrétním revizním bodem na kufru Apache Mahout. Tento bod revize je po vydání 0.9.0, ale před vydáním 0.10.0. To poskytuje velké množství oprav chyb a funkčnívylepšení oproti verzi 0.9.0, ale poskytuje stabilní verzi funkce Mahout před úplným převodem na nový Mahout založený na Sparku v 0.10.0.

Revizní bod zvolený pro Mahout v HDP 2.3.x a 2.4.x je z větve "mahout-0.10.x" Apache Mahout, od 19 Prosinec 2014, revize 0f037cb03e77c096 v GitHubu.

V HDP-2.5.x a 2.6.x jsme z Mahoutu odstranili knihovnu "commons-httpclient", protože ji považujeme za zastaralou knihovnu s možnými problémy se zabezpečením a upgradovali Hadoop-Client v Mahoutu na verzi 2.7.3, stejnou verzi používanou v HDP-2.5. Výsledek:

-   Dříve zkompilované úlohy Mahout bude nutné znovu zkompilovat v prostředí HDP-2.5 nebo 2.6.

-   Existuje malá možnost, že některé úlohy Mahout může dojít k "ClassNotFoundException" nebo "nelze načíst třídy" chyby týkající se "org.apache.commons.httpclient", "net.java.dev.jets3t", nebo související třídy název předpony. Pokud k těmto chybám dojde, můžete zvážit, zda ručně nainstalovat potřebné nádoby v cestě třídy pro úlohu, pokud je riziko problémů se zabezpečením v zastaralé knihovně přijatelné ve vašem prostředí.

-   Existuje ještě menší možnost, že některé úlohy Mahout mohou narazit na selhání v mahoutových voláních kódu hbase-client do knihoven hadoop-common kvůli problémům s binární kompatibilitou. Bohužel neexistuje žádný způsob, jak tento problém vyřešit, kromě návratu k HDP-2.4.2 verzi Mahout, která může mít problémy se zabezpečením. Opět by to mělo být velmi neobvyklé a je nepravděpodobné, že by se to stalo v daném pracovním apartmá Mahout.

          SSL\_\_RSA\_S\_RC4\_128 MD5, SSL\_RSA\_\_S RC4\_128\_\_\_\_\_\_\_\_\_\_SHA, TLS\_RSA\_s\_AES 128 CBC SHA, SSL RSA S 3DES EDE CBC SHA

          SSL\_\_RSA\_S\_RC4\_128 MD5, SSL\_RSA\_\_S RC4\_128\_\_\_\_\_\_\_\_\_\_SHA, TLS\_RSA\_s\_AES 128 CBC SHA, SSL RSA S 3DES EDE CBC SHA

- **RangerUI: Escape z politiky podmínky textu zadaného ve formuláři zásad**
    
  **Ovlivněná součást:** Ranger

  tags.attds\[&\#x27;dsds \#&x27; \] \#x27;cssdfs&x27; =& \#
